Adding Response Codes

In OpenAPI, you define response codes under the responses section of each endpoint.

/logs:
  post:
    summary: Submit a new log entry
    responses:
      '201':
        description: Log entry created successfully
        content:
          application/json:
            schema:
              $ref: '#/components/schemas/LogEntryApiResponse'
      '400':
        description: Bad request - invalid input
        content:
          application/json:
            schema:
              $ref: '#/components/schemas/ProblemDetails'
      '401':
        description: Unauthorized - authentication required
        content:
          application/json:
            schema:
              $ref: '#/components/schemas/ProblemDetails'
      '403':
        description: Forbidden - insufficient permissions
        content:
          application/json:
            schema:
              $ref: '#/components/schemas/ProblemDetails'
      '413':
        description: Payload too large - log entry exceeds size limit
        content:
          application/json:
            schema:
              $ref: '#/components/schemas/ProblemDetails'
      '429':
        description: Too many requests - rate limit exceeded
        content:
          application/json:
            schema:
              $ref: '#/components/schemas/ProblemDetails'
      '500':
        description: Internal server error
        content:
          application/json:
            schema:
              $ref: '#/components/schemas/ProblemDetails'
      '503':
        description: Service unavailable - logging system temporarily down
        content:
          application/json:
            schema:
              $ref: '#/components/schemas/ProblemDetails'

Key Points:

  • Use appropriate HTTP status codes (201 for created, 400 for bad request, etc.)
  • Provide a clear description for each response
  • Reference schemas for consistent response formats
  • Include both success and error responses

Adding Query Parameters

Query parameters are defined in the parameters section of an endpoint.

/logs:
  get:
    summary: Retrieve log entries
    parameters:
      - name: level
        in: query
        description: Filter by log level
        required: false
        schema:
          type: string
          enum: [DEBUG, INFO, WARN, ERROR, FATAL]
      - name: application
        in: query
        description: Filter by application name
        required: false
        schema:
          type: string
      - name: from
        in: query
        description: Start timestamp for filtering (ISO 8601 format)
        required: false
        schema:
          type: string
          format: date-time
      - name: to
        in: query
        description: End timestamp for filtering (ISO 8601 format)
        required: false
        schema:
          type: string
          format: date-time
      - name: page
        in: query
        description: Page number for pagination
        required: false
        schema:
          type: integer
          default: 1
          minimum: 1
      - name: size
        in: query
        description: Number of items per page
        required: false
        schema:
          type: integer
          default: 50
          minimum: 1
          maximum: 1000
      - name: sort
        in: query
        description: Sort order (field:direction)
        required: false
        schema:
          type: string
          default: "timestamp:desc"
    responses:
      '200':
        description: Successful operation
        content:
          application/json:
            schema:
              $ref: '#/components/schemas/LogEntryApiResponse'

              schema:
                $ref: '#/components/schemas/ProblemDetails'
    get:
      summary: Retrieve log entries
      parameters:
        - name: level
          in: query
          description: Filter by log level
          required: false
          schema:
            type: string
            enum: [DEBUG, INFO, WARN, ERROR, FATAL]
        - name: application
          in: query
          description: Filter by application name
          required: false
          schema:
            type: string
        - name: from
          in: query
          description: Start timestamp for filtering (ISO 8601 format)

          schema:
            type: string

Implementation Steps

  1. Open your OpenAPI specification file (usually a .yaml or .json file)
  2. Locate the endpoint you want to enhance (e.g., /logs)
  3. Add the parameters section for query parameters
  4. Expand the responses section with additional HTTP status codes
  5. Define or reference schemas for request and response bodies
  6. Test your updated specification using Swagger UI or other tools

Remember:

  • Your API implementation code doesn't change - only the documentation
  • Use appropriate HTTP status codes for different scenarios
  • Provide clear descriptions for all parameters and responses
  • Reference schemas to maintain consistency
  • Add examples to make your documentation more useful
