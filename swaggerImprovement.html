What to Add to Your Current Swagger

1. Add Detailed Response Codes

Your current documentation only mentions 200, 400, and 500 responses. You should add more specific response codes:

POST /logs

Response Codes: 201 Created (instead of 200) 400 Bad Request 401 Unauthorized 403 Forbidden 413 Payload Too Large 429 Too Many Requests 500 Internal Server Error 503 Service Unavailable

2. Add Request and Response Examples

Include examples for both requests and responses:

// Example Request { "timestamp": "2023-08-25T13:45:30Z", "level": "INFO", "message": "User login successful", "application": "online-banking-web", "userId": "user-12345", "correlationId": "corr-67890" } // Example Response { "id": "log-12345", "status": "accepted", "timestamp": "2023-08-25T13:45:30Z" }

3. Add Query Parameters for GET /logs

Your GET /logs endpoint should support filtering:

Parameter Type Description
level string Filter by log level (DEBUG, INFO, WARN, ERROR, FATAL)
application string Filter by application name
from string Start timestamp (ISO 8601 format)
to string End timestamp (ISO 8601 format)
page integer Page number for pagination
size integer Number of items per page

4. Define Complete Schemas

Your schemas should include all properties with descriptions:

LogEntry: type: object required: - timestamp - level - message - application properties: id: type: string description: Unique identifier for the log entry timestamp: type: string format: date-time description: Timestamp when the event occurred level: type: string enum: [DEBUG, INFO, WARN, ERROR, FATAL] description: Log severity level message: type: string description: Descriptive message of the logged event application: type: string description: Application or service name generating the log userId: type: string description: ID of the user associated with the event correlationId: type: string description: Correlation ID for tracing related events additionalData: type: object description: Additional structured data relevant to the event
